# 🚀 [TIL] 부록. 협업의 품격을 높여주는 커밋 컨벤션(Conventional Commits) 정리

## 1. 커밋 컨벤션, 왜 지켜야 할까?
혼자 코딩할 때는 "수정", "완료"라고만 적어도 알아봤지만, SSAFY에서 팀 프로젝트를 해보니 커밋 메시지만 보고도 **'이 사람이 지금 뭘 한 건지'** 바로 알 수 있게 하는 약속이 꼭 필요하다는 걸 깨달았다. 
일종의 개발자들 사이의 **'말머리'** 약속이라고 이해하면 쉽다.

---

## 2. 내가 실무에서 제일 많이 쓸 것 같은 TOP 6 (빈도순)

공부해보니 수많은 타입이 있지만, 아래 6가지만 제대로 써도 90%는 해결될 것 같다.

### 🌟 핵심 타입 정리
1. **`feat` (Feature): 새로운 기능을 만들었을 때**
   - 예: `feat: 로봇 팔 상하 이동 로직 추가`
   - 기능이 새로 생기면 무조건 이거다.
2. **`fix` (Bug Fix): 고장 난 걸 고쳤을 때**
   - 예: `fix: 센서 값이 튀는 현상 필터링 수정`
   - 버그를 잡았을 때 쓴다.
3. **`docs` (Documentation): 문서 관련 작업**
   - 예: `docs: Git 기초 가이드 Step 1~5 TIL 작성`
   - 지금 내가 하고 있는 마크다운 정리 작업이 바로 이거다!
4. **`refactor` (Refactoring): 코드를 깔끔하게 다듬었을 때**
   - 예: `refactor: 중복되는 모터 제어 함수 정리`
   - 결과는 똑같은데 코드만 보기 좋게 고쳤을 때 사용한다.
5. **`style` (Style): 로직은 그대로, 형식만 바꿨을 때**
   - 예: `style: 변수명 오타 수정 및 들여쓰기 정리`
   - 세미콜론을 빠뜨렸거나 단순 오타를 고쳤을 때 적절하다.
6. **`chore` (Chore): 자질구레한 설정 변경**
   - 예: `chore: .gitignore에 임시 로그 폴더 추가`
   - 코드랑은 직접 상관없는 설정 파일 등을 만질 때 쓴다.

## 3. 커밋 메시지 작성하는 나만의 공식

단순히 제목만 적는 게 아니라, 나중에 내가 봐도 이해하기 쉽게 3단 구성을 지키기로 했다.

```text
타입: 제목 (필수! 한 눈에 들어오게)

본문 (선택 - 왜 이렇게 고쳤는지 이유를 적으면 더 좋다)

꼬리말 (선택 - 관련된 이슈 번호가 있다면 기록)
```

**실전 적용 예시:**
```text
feat: 초음파 센서 거리 측정 기능 추가

- 로봇 전방 30cm 이내 장애물 감지 로직 구현
- 측정 주기를 100ms에서 50ms로 단축하여 반응 속도 개선
```

---

## 💡 학습 후기 및 꿀팁
- **소문자로 시작하기:** `Feat:` 보다는 `feat:` 처럼 소문자로 쓰는 게 관례라고 한다.
- **한 번에 한 가지만:** 기능을 추가하면서 오타까지 고쳤다면? 커밋을 두 번으로 나눠서 남기는 습관을 들여야겠다. 그래야 나중에 문제가 생겨도 되돌리기 편하다.
- **영어 vs 한국어:** 팀마다 다르겠지만, 일단 나는 공부하는 단계니 제목은 명확하게 한국어로 적고 타입만 영문 약속을 지키기로 했다.
