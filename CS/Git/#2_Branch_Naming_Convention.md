# 🚀 [TIL] 브랜치에도 이름표가 있다! 브랜치 명명 규칙 (Branch Naming Convention)

커밋 메시지에 규칙이 있듯, 브랜치도 `type/기능명` 형태의 이름표를 붙여 관리한다. 이렇게 하면 수십 개의 브랜치가 쌓여도 길을 잃지 않을 수 있다.

---

## 1. 브랜치 이름의 기본 구조

보통 **`타입/간결한-설명`** 형식을 사용한다. 
- 모든 글자는 **소문자**로 적는 것이 관례다.
- 띄어쓰기 대신 **하이픈(`-`)**이나 **언더바(`_`)**를 사용하여 연결한다. (보통 하이픈을 더 선호한다.)

---

## 2. 주로 사용하는 브랜치 타입 (Prefix)

Conventional Commits와 거의 비슷해서 금방 익숙해질 수 있다.

### 🌟 가장 많이 쓰는 3대장
1. **`feat/` (또는 `feature/`): 새로운 기능을 개발할 때**
   - 예: `feat/motor-control`, `feat/obstacle-avoidance`
   - 로봇 프로젝트의 핵심 기능 단위로 브랜치를 나눌 때 쓴다.
2. **`fix/` (또는 `bugfix/`): 버그를 수정할 때**
   - 예: `fix/sensor-error`, `fix/connection-timeout`
   - 잘못 돌아가는 코드를 고치기 위한 전용 브랜치다.
3. **`docs/` (Documentation): 문서 작업만 할 때**
   - 예: `docs/readme-update`, `docs/api-manual`
   - 코드 수정 없이 문서만 다듬을 때 사용한다.

### 🛠️ 실무에서 쓰는 추가 타입
4. **`refactor/`:** 코드 구조를 개선할 때 (결과는 같으나 코드만 깔끔하게)
   - 예: `refactor/clean-code`
5. **`test/`:** 테스트 코드를 작성하거나 수정할 때
   - 예: `test/unit-test-motor`
6. **`hotfix/`:** (매우 중요) 메인 브랜치에서 급하게 고쳐야 하는 치명적인 에러가 발생했을 때
   - 예: `hotfix/emergency-stop-error`

---

## 3. 대표적인 브랜치 전략: Git Flow vs GitHub Flow

브랜치를 어떻게 운영할지에 대한 '전략'도 공부했다.

### ✅ GitHub Flow (심플함 - SSAFY/개인 프로젝트 추천)
- **`main` 브랜치:** 언제나 실행 가능하고 배포 가능한 상태여야 한다.
- **`feature/` 브랜치:** 기능을 만들 때마다 생성하고 작업 후 PR을 통해 `main`에 합친다.
- 구조가 단순해서 나 같은 초보자나 작은 팀이 쓰기에 아주 좋다.

### ✅ Git Flow (체계적 - 실무/대규모 프로젝트용)
- `master`, `develop`, `feature`, `release`, `hotfix` 등 브랜치를 아주 세밀하게 나눠서 관리한다.
- 복잡하지만 그만큼 대규모 소프트웨어를 안정적으로 관리할 수 있다.

---

## 💡 영현's 실전 브랜치 꿀팁
- **동사보다는 명사로:** `feat/make-motor-move` 보다는 `feat/motor-control` 처럼 명사 위주로 짓는 게 더 깔끔하다.
- **이슈 번호 활용:** 팀 프로젝트에서는 이슈 번호를 붙이기도 한다. (예: `feat/12-motor-control`) 이렇게 하면 깃허브 이슈와 바로 연결되어 관리가 편하다.
- **삭제는 미루지 말자:** `main`에 성공적으로 병합된 브랜치는 아까 배운 명령어로 바로바로 지워주는 게 'Git 정리'의 핵심이다!

---

## 🎉 정리하며
커밋 메시지(`docs: ...`)와 브랜치 이름(`docs/...`)을 통일감 있게 가져가니까 협업의 전체적인 그림이 훨씬 선명해졌다. 이제 `Office-Guardian` 리포지토리에 브랜치 하나를 만들더라도 근거 있게 이름표를 붙일 수 있을 것 같다. 🔥
